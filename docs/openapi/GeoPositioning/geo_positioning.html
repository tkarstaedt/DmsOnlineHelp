Developer Documentation: OpenAPI GeoPositioning

Technical integration of vehicle and positioning data into the PEAK.X platform (e.g., PEAK.DMS).

1. Introduction

1.1 Purpose of the Document

This document describes the REST API GeoPositioning of the PEAK.X platform. It serves as a technical foundation for integrating vehicle and positioning data (GPS) in real-time.

1.2 Target Audience

  • Software Developers: Integrating vehicle tracking systems.
  • System Integrators: Coupling backend systems.

2. API Overview

The GeoPositioning API allows external systems to transmit current position data of mobile units (e.g., vehicles) to PEAK.X.

Use Cases

  • Transmission of real-time positions (GPS).
  • Provisioning for PEAK.ITCS & Control Centers.
  • Historical analysis of vehicle movements.

Architecture

Client → REST API → PEAK.X Backend → Topic System.

Synchronous REST call, internal asynchronous processing.

3. Technical Specifications

Base URL

https://<server>:5601/api/v1/common/localization/positioning-device/geopositions

Authentication

  • Basic Auth: Base64 encoded in Header.
  • Role: GeoPositionsDmsOpenApi PutGeoPosition required.

Rate Limiting

  • PermitLimit: 100 Requests / 10s
  • QueueLimit: 2 Requests
Important: Communication exclusively via HTTPS. Timestamps must be in UTC (ISO 8601).

4. Endpoint Description

PUT /api/v1/common/localization/positioning-device/geopositions

Transmits a list of geoposition records to the system.

Request Body (JSON Array)

[
  {
    "deviceId": "4711",
    "deviceType": "PositioningDevice",
    "coordinateNorth": 41.40338,
    "coordinateEast": 2.17403,
    "altitude": 43.5,
    "azimuth": 243.5,
    "coordinateSystem": "WGS84",
    "timeOfPosition": "2025-01-01T10:00:00Z",
    "accuracy": 4.3,
    "velocity": 55.6,
    "receivedSignalStrengthIndicator": -55.6,
    "timestamp": "2025-01-01T10:00:01Z"
  }
]

Response (202 Accepted)

{
  "status": 202,
  "message": "Position data successfully accepted."
}

5. Data Model: GeoPosition

Detailed description of fields in the JSON object.

Field Type Mandatory Description
deviceId string Yes ID of the sending device.
deviceType string Yes Device type (e.g., GPS Module).
coordinateNorth number Yes Latitude (WGS84) or North value (ENU).
coordinateEast number Yes Longitude (WGS84) or East value (ENU).
coordinateSystem enum Yes WGS84 or ENUInMeters.
timeOfPosition datetime Yes Time of measurement (UTC ISO 8601).

* For optional fields (altitude, velocity, etc.) see Swagger definition.

6. Examples & Errors

400 Bad Request

Occurs when mandatory fields are missing or data types are incorrect.

{
  "status": 400,
  "title": "Bad Request",
  "detail": "Field 'coordinateSystem' is missing."
}

8. Best Practices

  • HTTPS only: Security is the top priority.
  • Retry Logic: Implement backoff strategies for HTTP 429 (Rate Limits).
  • Time Sync: Regularly check system time (UTC).
